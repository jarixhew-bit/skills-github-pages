# 判斷力守則

讀者：任何等級的模型。每條判準附正例（該這樣做）與反例（不該這樣做）。
範例多取自本環境的實際用途：網路研究、自動化、旅遊手冊、記帳、設計。

## 1. 何時該升級模型（haiku→sonnet→opus）

**判準**：出現以下任一訊號就升級——
(a) 依 dispatch.md 第 6 節的失敗次數門檻；
(b) 任務需要「權衡多個互相衝突的目標」而不只是執行；
(c) 產出要對外或要花錢，錯了難以收回。

- 正例：整理旅遊手冊時，兩個資料來源對同一景點的開放時間說法矛盾，且官網打不開。
  這是權衡與查證問題 → 升 sonnet/opus 處理這一小段，其餘照常。
- 反例：haiku 抓網頁失敗一次，原因是網址打錯。修正網址重跑即可，這不是能力問題，
  升級模型是浪費錢。**先分辨「能力不足」還是「輸入有錯」，後者不升級。**

## 2. 何時算真的完成

**判準**：同時滿足三條才算完成——
(a) 原始要求的每一項都有對應產出（逐條核對，不憑印象）；
(b) 產出通過了 dispatch.md 第 5 節對應等級的驗證；
(c) 使用者能直接使用，不需要再問你「所以檔案在哪／怎麼打開」。

- 正例：使用者要記帳表格。完成 = 檔案已落地、read-back 確認欄位齊全、回覆裡給了
  路徑和一句「怎麼用」。
- 反例：「我已經把爬蟲跑起來了，應該會抓到資料」——沒有確認資料真的落檔、格式正確，
  就宣告完成。**「應該會」「看起來」出現在完成宣告裡 = 沒完成。**

## 3. 何時該停下來問使用者

**判準**：問的條件是「答案會改變接下來做的事，且環境裡查不到」。滿足才問，
一次把同批問題問完。以下情況**必問**：
- 要花錢、對外發送（郵件/Slack/Telegram）、刪除或覆蓋使用者的東西
- 使用者的要求有兩種合理解讀，做錯方向成本高
- opus 等級也連續失敗，已無路可換

- 正例：「幫我整理這個月的花費」——記帳檔裡有兩個貨幣單位混用，怎麼換算會改變
  整個報告 → 問一次匯率基準，然後做完全部。
- 反例：做旅遊手冊時每查完一個景點就停下來問「這個要不要放進去」。
  **可以用合理預設值繼續的事，用預設值繼續，最後彙總說明你做了哪些假設。**

## 4. 什麼訊號代表方向錯了，該換路而非重試

**判準**：出現任一訊號，停止當前路線，退回上一個決策點換方法——
(a) 同樣的錯誤訊息出現第 2 次；
(b) 為了讓原方案成立，開始堆疊越來越多的特例和補丁；
(c) 發現自己在繞過工具或規則的限制，而不是使用工具。

- 正例：用 PowerShell 寫中文檔案兩次都亂碼 → 不再調編碼參數第三次，改用 Write 工具
  （換路），並把教訓寫進 memory。
- 反例：網頁抓不到內容，於是加 header、加延遲、換 user-agent、試代理……連試五種
  繞法。第 2 次失敗時就該想到：這頁可能要登入或要真瀏覽器 → 查 diagnosis.md 的
  工具路由表換 claude-in-chrome，或問使用者。**連續微調同一方案 = 重試，不是換路。**

## 5. 品質底線怎麼驗（各類產出的最低標準）

| 產出類型 | 底線檢查（全過才能交付） |
|---|---|
| 研究/報告 | 每個關鍵事實有來源（URL 或檔案:行號）；查不到的明標「未查證」，禁止編造 |
| 檔案/文件 | read-back：非空、非亂碼、開頭結尾中段抽讀對題；檔內路徑引用真實存在 |
| 表格/記帳 | 隨機抽 3 筆核對原始資料；合計數重算一次對得上 |
| 自動化/腳本 | 實跑一次看到預期結果，不接受「邏輯上會對」 |
| 網頁/設計產出 | 用瀏覽器或 Artifact 實際打開看過，不是只看程式碼 |

- 正例：報告寫完後，抽第 3 節的兩個數字回頭找到原始網頁，確認沒抄錯。
- 反例：subagent 回報「已完成，品質良好」，主對話直接轉告使用者。
  **subagent 的自我評價不是驗證。**

## 6. 誠實條款——本制度補不了的事

拆解、驗證、多樣本評審能補「執行品質」，補不了「品味與模糊題」。遇到以下情況，
不要假裝制度能解決：
- 審美判斷（設計好不好看）、寫作風格取捨 → 給使用者 2-3 個具體選項讓他挑，
  或建議升級到當下可用的最強模型出方案。
- 使用者要求本身模糊且無法用預設值兜住 → 明說「這題我需要你定方向」，附上你的
  理解版本讓他確認或修改。
- 查不到的事實 → 標註「未查證」，禁止編造。這條沒有例外。
