# 派工 Prompt 模板（可攜版）

用法：挑對應任務型態的模板，把〔〕填滿後作為 Agent 呼叫的 prompt。
填不出來就看同目錄 `examples.md`——有三個填好的完整範例（研究/改頁/驗收），照著改詞即可。
每份模板都已內建派工三件套（目標與動機／驗收條件／回報格式）。
`model` 與 `subagent_type` 依 dispatch.md 第 4 節選。**空著〔驗收條件〕不填 = 不准派工。**

## 通用回報合約（每份模板結尾都要帶，已內嵌）

> 回報要求：只回結論。(1) 結論 5 行以內；(2) 證據用 檔案路徑:行號 或 URL；
> (3) 未解決事項與原因。長產物寫到〔落檔路徑〕，回報只給路徑。
> 禁止貼回大段原文。

---

## 模板 1：搜尋（agent: Explore，model: haiku）

```
目標：找出〔要找的東西：檔案/設定/某段文字〕。
動機：〔為什麼要找，讓你能判斷相近結果算不算〕。
範圍：〔目錄或線索〕。廣度：〔quick/medium/very thorough〕。
驗收條件：
- 找到則列出完整路徑（含行號，如適用）
- 找不到則列出實際搜過的位置與用過的關鍵字，證明搜索足夠廣
回報要求：（通用回報合約）
```

## 模板 2：實作（agent: general-purpose，model: sonnet）

```
目標：做出〔具體產出，例如：東京五日遊手冊的交通章節，markdown 格式〕。
動機：〔給誰用、解決什麼問題〕。
輸入材料：〔路徑或 URL 清單〕。
落檔路徑：〔明確路徑〕。
驗收條件：
- 〔內容必須涵蓋的項目，逐條列〕
- 檔案已落地、非空、格式為〔格式〕
- 每個事實性陳述附來源；查不到的標「未查證」
限制：〔不准動的東西／字數或範圍上限〕。
（若成品是給使用者看的視覺化產出：subagent 落檔後，由主對話用 Artifact 工具呈現，
這一步不委派。）
回報要求：（通用回報合約）
```

## 模板 3：重構／批次修改（agent: general-purpose；模式已定型用 haiku，未定型用 sonnet）

```
目標：把〔範圍內的檔案〕按以下模式修改：〔修改前→修改後的具體範例，至少一組〕。
動機：〔為什麼改〕。
範圍：〔明確檔案清單或 glob〕。
驗收條件：
- 範圍內每個檔案都已處理，列出已改清單
- 不改範圍外的任何檔案
- 備份：git 管理下的檔案改前確認 git status 乾淨即可；非 git 管理的檔案先複製到
  〔備份路徑，本機預設 ~/.claude/backups/〕
- 抽 2 個改完的檔案在回報中附 修改處的檔案路徑:行號
回報要求：（通用回報合約）
```

## 模板 4：研究（agent: general-purpose，model: sonnet）

```
目標：回答〔具體問題，可判定答沒答到的那種〕。
動機：〔這答案要拿來做什麼決定〕。
方法提示：先 WebSearch 找 3-5 個來源，交叉比對；來源矛盾時兩說並陳。
落檔路徑：〔研究筆記寫到哪〕。
驗收條件：
- 問題有直接答案（或明確說「查不到」＋查過哪裡）
- 每個關鍵結論至少 1 個來源 URL，重要數字要 2 個獨立來源
- 區分「來源說的」與「你推論的」
回報要求：（通用回報合約）
```

## 模板 5：審查／驗收（agent: verifier，model: sonnet；verifier 不可用時見 dispatch.md 第 0 節備援）

```
待驗物：〔路徑或位置〕。
背景：這是〔誰〕為了〔目標〕做的，你沒看過過程，請以全新視角檢查。
原始驗收條件：
- 〔把當初派工時的驗收條件原樣貼上，逐條〕
額外要求：依你的 agent 定義檢查檔案存在性、引用真實性、規則矛盾、模糊語句。
回報要求：第一行「通過」或「不通過（N 個缺陷）」，缺陷每條一行附位置。
```

## 使用備註

- **混合型任務要串接，不要硬塞一個模板**。例如「研究沖繩親子景點並寫成手冊」：
  先用模板 4 派研究（產出＝研究筆記檔），再用模板 2 派成文（輸入材料＝那份筆記），
  最後主對話 read-back 或派模板 5 驗收。一個 subagent 同時研究＋成文，容易兩頭都淺。
- 高風險判斷要第二意見時：把模板 4 同一題派給兩個獨立 agent，答案不一致 → 升 opus
  裁決或問使用者（dispatch.md 第 5 節）。
- 升級重派時（dispatch.md 第 6 節），在模板前面加一段：
  「前次嘗試失敗軌跡：〔做了什麼／錯誤訊息原文／為什麼失敗〕。請避開同樣的路。」
