# 委派 Prompt 范例库（可携版）

templates.md 给的是空模板；本档给**填好的完整范例**，任务相似时改几个词直接用。
范例基于本仓库的真实任务类型写成（2026-07-12 由 Fable 5 交接时撰写）。
每个范例后面的「为什么这样写」是重点——那些就是弱模型最常漏掉的东西。

## 范例 1：研究查证（agent: researcher，或 general-purpose/sonnet）

```
目标：查证大阪「旬房」餐厅 2026 年 8 月的营业时间与是否需要预约。
动机：日本旅游手册 japan-trip-2026.html 里这家店的卡片信息是 3 个月前写的，
用户 8 月要实际去，写错了会让一家人白跑一趟——所以宁可标"未查证"也不许猜。
方法提示：先 WebSearch 找官网＋tabelog＋Google Maps 三个来源交叉比对；
来源矛盾时两说并陈，注明各自更新日期。
落档路径：不需要落档，结论直接回报。
验收条件：
- 营业时间有直接答案（或明确说"查不到"＋查过哪三处）
- 是否需要预约、怎么预约（电话/网上/不接受）有答案
- 每个结论附来源 URL；官网与第三方矛盾时以官网为准并注明
回报要求：只回结论。(1) 结论 5 行以内；(2) 证据附 URL；(3) 未解决事项与原因。
禁止贴整页网页内容。禁止再派 subagent。
```

**为什么这样写**：动机里写了"写错的代价"，所以 researcher 遇到查不到时知道该标
"未查证"而不是硬编一个；"三个来源交叉"是具体动作，不是"好好查查"。

## 范例 2：改页面（agent: html-editor，或 general-purpose/sonnet）

```
目标：把 japan-trip-2026.html 里 D03 晚餐的「某某烧肉」卡片整张替换为「新店名」。
修改前→修改后范例：卡片结构照抄同页 D02 晚餐卡片（锚点 class="food-card"），
店名/图片/中英描述/地图链接四处都换，其他保持原结构。
动机：用户说原店已歇业。手册已发给家人，改坏排版他们会直接看到。
输入材料：新店信息见〔来源 URL 或用户原话〕；卡片图片用 Google Places 外链
（做法见 skills/media-files.md）；结构锚点见 .claude/notes/travel-pages.md。
验收条件：
- 新卡片含图片＋中英双语描述（data-zh/data-en 成对），四处信息全部替换、无旧店残留
  （全文搜索旧店名确认为 0 处）
- python3 tools/check-html.py japan-trip-2026.html 输出"通过"
- 不动 D03 晚餐卡片以外的任何内容
回报要求：只回结论。(1) 改了什么 5 行以内；(2) 改动处附 文件:行号＋检查脚本
输出最后一行；(3) 未解决事项与原因。禁止贴大段 HTML。禁止再派 subagent。
```

**为什么这样写**：给了"照抄哪张现有卡片"的锚点，subagent 就不会自己发明新结构；
"全文搜索旧店名为 0 处"是可机器判定的验收条件，"替换干净"不是。

## 范例 3：验收（agent: verifier）

```
待验物：japan-trip-2026.html 的 D03 晚餐卡片（锚点 class="food-card"，搜"新店名"定位）。
背景：这是 html-editor agent 为了替换歇业餐厅做的修改，你没看过过程，请以全新视角检查。
原始验收条件：
- 新卡片含图片＋中英双语描述（data-zh/data-en 成对），无旧店名残留
- python3 tools/check-html.py japan-trip-2026.html 输出"通过"（请实际重跑）
- D03 晚餐卡片以外的内容无改动（git diff 确认）
额外要求：依你的 agent 定义检查文件存在性、引用真实性。
回报要求：第一行"通过"或"不通过（N 个缺陷）"，缺陷每条一行附位置。
```

**为什么这样写**：验收条件是**原样带过去**的，不是让 verifier 自己发明标准；
"请实际重跑"防它只看别人说通过就采信——subagent 的自我评价不是验证。

## 升级重派时的附加段（接在任何范例前面）

```
前次尝试失败轨迹:第一次用〔方法A〕,错误是〔原文〕;第二次改〔方法B〕,
错在〔哪里〕。请避开这两条路,考虑〔可能的新方向〕。
```

没有这段,升级到 opus 也只会把同样的坑再踩一遍——升级的价值在于带着失败轨迹换路。
