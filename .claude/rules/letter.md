## 使用者沒問、但你最需要知道的三件事

### 1. 開場健檢〔本機限定〕

- 本機 `~/.claude/settings.json` 的 `model` 曾被設為 `claude-fable-5[1m]`。此模型是
  限期提供的，失效後 session 可能報錯或行為異常。若使用者反映「怪怪的」或你發現
  模型設定仍是 fable，提醒他跑 `/model` 選 Sonnet。
- 同一個 settings.json 裡有**明文的 Google Maps API key**。key 若外流會被盜用計費。
  觸發時機：使用者下次問到設定、安全，或你下次需要動 settings.json 時，提一次
  「值得在 Google Cloud 後台加用量上限，也可研究改放環境變數（派 claude-code-guide
  查官方寫法）」；提過一次就寫進 memory，之後不再重複。
- 本機啟用的 plugin 很多（11 個啟用、1 個已停用），大部分用不到卻佔工具空間。
  觸發時機：使用者抱怨變慢或工具混亂時，建議他用 `/plugin` 檢視停用。停用前務必
  先問使用者，這是他的設定。

### 2. 使用者畫像——決定你該怎麼說話和交付（所有環境適用）

YANG 是**非工程師的新手使用者**：中文溝通、用途是生活型任務（旅遊手冊、記帳、
設計、網路研究、日常自動化）。這意味著：
- 交付標準是「能直接用」：檔案給路徑＋一句怎麼打開；視覺產出用 Artifact；
  不要交一堆需要他自己組裝的半成品。
- 記帳、金額、日期這類數字，錯一個就毀掉信任——一律按 judgment.md 第 5 節抽驗。
- 他曾嘗試安裝第三方 multi-agent 框架（AI Team OS），說明他想要「一隊 AI 幫他做事」。
  本制度（dispatch.md 的派工體系）就是原生版本，不需要第三方框架——他要類似東西時，
  先展示原生 subagent 能做到什麼。

### 3. 制度不會自己執行——你要主動接上它（所有環境適用）

CLAUDE.md 每次都會載入，但只是路由表。它有用的前提是你**真的照路由去讀檔**：
派工前讀 dispatch.md，拿不準時讀 judgment.md。學到教訓照 maintenance.md 第 3 節
落地（本機寫 memory＋rules；雲端寫 rules 並 commit）。這套制度最大的敵人不是
規則錯，是規則被靜靜略過。

## 這套制度最可能的退化方式與預防

| 退化方式 | 徵兆 | 預防 |
|---|---|---|
| 規則檔膨脹成沒人讀的長文 | 單檔超過 150 行 | maintenance.md 第 4 節的精簡門檻，認真執行 |
| 派工流於形式 | 委派 prompt 沒填驗收條件 | templates.md 開頭鐵律：不填不准派 |
| 路由失效 | CLAUDE.md 指到不存在的檔 | 不要改 rules/ 檔名；精簡時跑 audit（maintenance 第 5 節） |
| 教訓紀錄變垃圾場 | 條目重複、互相矛盾 | 滿 10 條就精簡，矛盾派 verifier 查 |
| 規則都遵守但精神丟失 | 驗證變成「掃一眼算 read-back」 | 驗證用可檢查的底線表（judgment 第 5 節），重要產出必派獨立 verifier |
| **兩地版本分岔**（可攜版新增） | 本機與雲端的 rules 內容不一致 | 開工先 `git pull`、改完必 `git push`（maintenance 第 1 節）；發現分岔時以較新 commit 為準合併 |

## Harness 極限的誠實標註（建立時已知，補不了的）

- 拆解、驗證、多樣本評審補得了執行品質；**品味與模糊題補不了**。遇到時照
  judgment.md 第 6 節：給選項、要求定方向、或明說做不到。
- Agent 呼叫不能逐次指定 effort（只有 model）；effort 走 agent 定義檔的 `effort` 欄位。
- 自動 compaction 可能吃掉對話早期的指令。長任務要把關鍵約束落檔（如 todo 檔），
  別只留在對話裡。
- memory 系統只存在本機；雲端 session 的持久化只有這個 repo 本身。
