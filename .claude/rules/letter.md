## 這套制度最可能的退化方式與預防

| 退化方式 | 徵兆 | 預防 |
|---|---|---|
| 規則檔膨脹成沒人讀的長文 | 單檔超過 150 行 | maintenance.md 第 4 節的精簡門檻，認真執行 |
| 派工流於形式 | 委派 prompt 沒填驗收條件 | templates.md 開頭鐵律：不填不准派 |
| 路由失效 | CLAUDE.md 指到不存在的檔 | 不要改 rules/ 檔名；精簡時跑 audit（maintenance 第 5 節） |
| 教訓紀錄變垃圾場 | 條目重複、互相矛盾 | 滿 10 條就精簡，矛盾派 verifier 查 |
| 規則都遵守但精神丟失 | 驗證變成「掃一眼算 read-back」 | 驗證用可檢查的底線表（judgment 第 5 節），重要產出必派獨立 verifier |
| **兩地版本分岔**（可攜版新增） | 本機與雲端的 rules 內容不一致 | 開工先 `git pull`、改完必 `git push`（maintenance 第 1 節）；發現分岔時以較新 commit 為準合併 |

## Harness 極限的誠實標註（建立時已知，補不了的）

- 拆解、驗證、多樣本評審補得了執行品質；**品味與模糊題補不了**。遇到時照
  judgment.md 第 6 節：給選項、要求定方向、或明說做不到。
- Agent 呼叫不能逐次指定 effort（只有 model）；effort 走 agent 定義檔的 `effort` 欄位。
- 自動 compaction 可能吃掉對話早期的指令。長任務要把關鍵約束落檔（如 todo 檔），
  別只留在對話裡。
- memory 系統只存在本機；雲端 session 的持久化只有這個 repo 本身。

## Fable 5 最後交接（2026-07-12 補記）

使用者問「還能做什麼把 Fable 5 的能力搬給 Sonnet/Opus」，這次交接工程的回答是
四樣東西，之後的你要**主動用上它們**，否則等於沒搬：

1. **檢查已腳本化**：`tools/check-html.py`（標籤平衡＋siteLangUser key 檢查），
   並掛了 CI（`.github/workflows/html-check.yml`）在每個 PR 自動跑。改 HTML 後
   自己先跑，別等 CI 紅了才修。
2. **結構筆記**：`.claude/notes/` 每個專案一份——改頁面前先讀對應筆記，按錨點
   定位，不要重新通讀整個檔案。筆記過時了就順手更新它（這是筆記存活的唯一方式）。
3. **專屬 agent**：`researcher` 與 `html-editor`（連同原有 `verifier`），方法論
   與高 effort 已燒進定義檔。常設任務優先派它們，見 dispatch.md 第 0 節。
4. **範例庫**：`.claude/rules/examples.md`，委派 prompt 照範例改詞，重點看每個
   範例後的「為什麼這樣寫」。

搬不動的東西（誠實條款，judgment.md 第 6 節）：品味、模糊需求的一次到位、跨領域
直覺。遇到這些，制度給的出路是：給使用者 2-3 個具體選項、或升級到當下可用的最強
模型。不要假裝文檔能替代能力——文檔能做的是讓你**少犯已犯過的錯、把粗活派出去、
在宣告完成前真的驗過**。做到這三件，Sonnet/Opus 跑這套制度的日常表現就足夠好。
