# 制度檔維護協議（可攜版）

管轄範圍：本 repo 的 `CLAUDE.md`、`.claude/rules/*.md`、`.claude/agents/*.md`。
讀者：未來想修改這些檔案的任何模型（本機或雲端）。

## 1. 改之前——備份與同步鐵律

- 本 repo 由 git 管理，**git 就是備份**：改前先 `git status` 確認工作區乾淨
  （有未 commit 的雜項先 commit 或問使用者），改完立刻 commit。
- 改前先 `git pull`——另一邊環境可能已更新過制度，別蓋掉別人的改動。
- 改完 `git push`。push 失敗（無權限/無網路）不阻擋工作，但要提醒使用者一句。
- repo 之外的全域檔（本機 `~/.claude/CLAUDE.md`）不在 git 管理下，改前先複製到
  `~/.claude/backups/rules/`，檔名加日期。**沒備份不准改。**

## 2. 什麼可以自行改、什麼要先問使用者

**可自行改（改完在回覆中告知一句即可）**：
- 修正確定的事實錯誤（路徑失效、工具改名、參數變更——需先查證屬實）
- 在「教訓紀錄」區追加條目（見第 3 節）
- 補充正例/反例、修正錯字

**必須先問使用者才能改**：
- 三條鐵律（CLAUDE.md）的內容或門檻數字（如「失敗 2 次」改成 3 次）
- 升降級路徑的模型選擇與次數門檻（dispatch.md 第 6 節）
- 刪除任何整節內容
- 新增會約束未來所有 session 的新規則（教訓紀錄除外）

**任何情況都不准做**：把制度檔內容複製進 CLAUDE.md（CLAUDE.md 只放鐵律與路由）；
在多個檔案寫重複規則（一條規則只住一個家，其他地方用引用）。

## 3. 踩坑教訓寫回哪裡、什麼格式

判斷順序：
1. **使用者的偏好或糾正** → 本機寫 memory（system prompt 有格式，type: feedback）；
   雲端沒有 memory，寫進本檔末尾「教訓紀錄」並 commit。
2. **環境事實**（工具行為、平台坑、網站特性）→ 寫進對應 rules 檔末尾的
   「## 教訓紀錄」區（沒有就建一個），標明適用環境（本機/雲端/皆是）。
3. **兩者皆是** → 兩邊都寫，memory 裡用一句話，rules 裡寫細節。

教訓條目格式（一條一行到三行）：
```
- [2026-07-03][本機/雲端/皆是] 情境：〔何時會遇到〕。教訓：〔該怎麼做〕。來源：〔踩了什麼坑〕。
```

## 4. 累積多長要精簡

- 任一檔案的「教訓紀錄」滿 **10 條**，或整個檔案超過 **150 行**：觸發精簡。
- 精簡做法：把重複的合併、已過時的刪除、已成通則的上升為該檔正文規則
  （上升到正文屬於「新增規則」，若涉及第 2 節的必問項要先問使用者）。
- 精簡前先 commit 當前狀態（等同備份）。

## 5. 防退化——每次精簡時順帶自查

精簡任何檔案時，順帶跑一次快速 audit：
- CLAUDE.md 路由表指的檔案是否都還存在（用 Glob 查 `.claude/rules/*.md`）
- 有沒有兩條規則互相矛盾（拿不準就派 verifier 查）
- 有沒有混進「保持高品質」這類無法執行的空話——刪掉或改寫成可檢查的條件

## 6. 改完之後

- read-back 驗證改動落地，然後 commit + push。
- 改動若涉及規則語義（不只是錯字），派 verifier 檢查該檔（模板 5）。
- 在回覆中用一兩句告訴使用者改了什麼、為什麼。
