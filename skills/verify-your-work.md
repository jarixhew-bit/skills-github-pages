# 技能：驗證自己的工作（宣告完成前的檢查）

## 目的
「我改好了」和「真的能用」之間的差距，是本專案最貴的錯誤來源。使用者是非工程師，他不會幫你發現半成品——他會直接把壞掉的網址發給別人。

## 何時觸發
每次準備說「完成」之前。沒有例外。

## 機械步驟（按產出類型選一行，全過才能交付）
| 產出類型 | 最低驗證動作 |
|---|---|
| HTML 頁面 | 用瀏覽器（playwright/chrome-devtools）實際打開，看到渲染結果；只看程式碼不算 |
| 檔案/文件 | read-back：讀回開頭、結尾、隨機中段各一處，確認非空、非亂碼、對題 |
| 腳本/自動化 | 實跑一次，看到預期輸出落地；「邏輯上會對」不算 |
| 數字（記帳/金額/日期） | 隨機抽 3 筆對原始資料，合計數重算一次（見 `skills/numbers-and-money.md`） |
| 部署上線 | 等 GitHub Pages 構建後，WebFetch 實際網址確認 200 且內容是新版 |

之後再核對兩件事：
1. 逐條核對原始要求——每一項都有對應產出，不憑印象。
2. 檢查自己的完成宣告裡有沒有「應該」「看起來」「大概」——有就代表沒驗完，回去驗。

## 硬規則
- **驗證不自驗**：重要或複雜產出（新頁面、批次改動、制度修改）另派 `verifier` agent（定義在 `.claude/agents/verifier.md`），附上原始驗收條件。verifier 不可用時派 general-purpose/sonnet 並把 verifier.md 規則貼進 prompt。
- subagent 回報「已完成、品質良好」不是驗證。主對話必須自己 read-back 或派 verifier。
- 驗證發現問題 → 修完後**重新走完整驗證**，不是只驗修的那一處。

## 本專案實例
制度檔建立後（2026-07-03），派 verifier 全面檢查，發現 CLAUDE.md 寫「制度正本在 workspace repo」與實際不符（正本已遷到本 repo），由 commit `659b564` 修正。做的人（當時的主對話）自己完全沒發現——因為它「知道自己想表達什麼」，只有沒看過過程的 fresh context 才會逐字讀出矛盾。這就是為什麼重要產出必須派獨立 verifier，而不是自己再看一遍。
