# 技能：倉庫與資料夾組織（東西放哪裡）

## 目的
「放哪裡」的錯誤幾乎不可逆：公開 repo 進了私密內容 = 已洩露；根目錄散了檔案 = 網址發出去後永遠搬不動。此檔是放置決策的查表。

## 何時觸發
建立任何新檔案、新專項之前；使用者說「存進 vault」「查 vault」時。

## 機械步驟（依序查表）
1. **私密內容？**（個人筆記、身份資料、任何使用者不想公開的東西）
   → 一律放 `jarixhew-bit/ai-vault`（獨立私有 repo，直推 main 不走 PR，管理規則見該 repo 自己的 CLAUDE.md）。網頁版先用 add_repo 把它加進 session；本地直接 clone/pull。
   → **本 repo 是公開倉庫，私密內容零容忍。** 拿不準算不算私密 → 當私密處理，問使用者。
2. **對外網頁 / App / 手冊？** → 本 repo，且**必須開獨立資料夾**（參照 `xisui/`、`trading/`）。
3. **制度規則、給 Claude 的工作指示？** → 本 repo `.claude/rules/`（唯一正本）。
4. **操作技能、交接文件？** → 本 repo `skills/`（本資料夾）。
5. **臨時檔、中間產物？** → scratchpad（session 專屬臨時目錄），不進任何 repo。

## 硬規則
- 新專項禁止把檔案散放根目錄。根目錄現有舊頁面（`fortune.html`、`expense-tracker.html` 等）是為保住已發出的網址而不搬，**它們是歷史包袱，不是可仿效的先例**。
- 分支不用於分類（合併即刪）；repo 之間不放制度副本（workspace repo 的副本已於 2026-07-03 作廢，教訓：兩份正本必然分岔）。
- 一條規則只住一個家：CLAUDE.md 只放鐵律與路由，細則住 rules/，其他地方用引用。發現同一規則寫在兩處 → 按 maintenance.md 處理成引用。

## 本專案實例
2026-07-07 建立 `ai-vault`：使用者要 AI 個人知識庫（第二大腦）。決策過程走的就是上面第 1 步——知識庫內容私密，而本 repo 公開，所以**開獨立私有 repo** 而不是在本 repo 開資料夾；又因為 vault 是單人筆記、無上線流程，所以直推 main 不走 PR（流程跟著風險走：公開頁面要 PR 驗證，私人筆記不用）。放置決策裡藏著兩個維度：**公開性決定放哪個 repo，風險決定走什麼流程。**
