# 技能：範圍控制（什麼在任務內、什麼不准碰）

## 目的
防止「順手多改」：順手重構、順手改排版、順手刪「看起來沒用」的檔案。這些行為在本專案造成過網址失效風險與 CLAUDE.md 與實際不符。

## 何時觸發
每次改檔前，以及過程中每次想動「計劃清單以外的檔案」時。

## 機械步驟
1. 開工時已列出要動的檔案清單（見 `skills/plan-before-work.md` 第 5 步）。
2. 過程中發現清單外的問題（錯字、壞連結、可疑的重複檔），套用判別式：
   - **會導致本次任務失敗** → 修，並在回覆中說明。
   - **不影響本次任務** → 不修。記一行到回覆末尾的「順帶發現」，讓使用者決定。
3. 以下東西任何任務都不准碰，除非使用者點名：
   - 根目錄的舊頁面檔名與路徑（網址已發出去，改名 = 別人手上的連結全斷）。
   - `.claude/rules/` 的規則語義（維護協議見 `.claude/rules/maintenance.md` 第 2 節）。
   - 別的專項資料夾（做 `trading/` 的任務不准動 `xisui/`）。
4. 刪除或覆蓋任何自己沒建立的檔案前：先讀它，確認內容與你以為的一致；不一致就停下來報告，不刪。

## 硬規則
- 「這個檔案看起來沒用」不是刪除理由。本 repo 有檔案是為保住網址而留的。
- 使用者要求有兩種解讀且做錯成本高 → 問一次再做（judgment.md 第 3 節）。
- 順帶發現彙總報告，但不順帶修。

## 本專案實例
`usj-disney-restaurants-v2.html` 目前存在於根目錄，但 CLAUDE.md 寫「v2 文件已刪除」。一個不守範圍的模型會在做別的任務時「順手」把它刪掉——但 git log 顯示它是 #87（`7ec6de4`）在「已刪除」的記載之後重新加回的，可能正在使用中。正確做法：不刪，在回覆裡寫一行「發現 v2 檔案與 CLAUDE.md 描述不符，要我處理嗎」，讓使用者裁決。刪錯一個上線頁面的代價遠大於多問一句。
