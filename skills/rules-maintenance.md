# 技能：維護制度檔與技能庫（怎麼安全地改規則）

## 目的
規則系統最大的敵人不是規則錯，是被靜靜略過或被改壞。此檔是改 `CLAUDE.md`、`.claude/rules/*.md`、`.claude/agents/*.md`、`skills/*.md` 時的操作流程摘要（完整協議在 `.claude/rules/maintenance.md`，衝突時以彼為準）。

## 何時觸發
想修改任何規則/技能檔時；學到新教訓要落檔時；發現兩條規則矛盾時。

## 機械步驟
1. **改前**：`git pull`（另一環境可能已改過）→ `git status` 確認乾淨。git 就是備份，不需另外複製。
2. **判斷許可權**（maintenance.md 第 2 節）：
   - 可自行改：修正查證屬實的事實錯誤、追加教訓紀錄、補例子、修錯字。改完回覆裡告知一句。
   - 必先問使用者：改三條鐵律或門檻數字、改升降級路徑、刪除整節、新增約束所有 session 的新規則。
   - 永遠禁止：把細則複製進 CLAUDE.md；同一規則寫進兩個檔案。
3. **教訓落檔格式**（一至三行）：
   `- [日期][本機/雲端/皆是] 情境：〔何時遇到〕。教訓：〔怎麼做〕。來源：〔踩了什麼坑〕。`
   寫進對應檔案末尾的「## 教訓紀錄」區。
4. **改後**：read-back 驗證 → commit + push（走功能分支＋PR；本地無 PR 工具可直推 main 但要說明）→ 涉及規則語義的改動派 verifier 檢查。
5. **精簡觸發線**：教訓滿 10 條或單檔超 150 行 → 合併重複、刪過時、通則上升為正文（上升屬「新增規則」要先問）。精簡時順帶 audit：路由指向的檔案都存在嗎（Glob 查）、有沒有互相矛盾、有沒有「保持高品質」這類不可執行的空話。

## 硬規則
- 沒 pull 就改 = 可能蓋掉另一環境的改動。這是兩環境共用 repo 的第一殺手。
- 改規則檔的 PR 不夾帶其他改動，讓 diff 一眼可審。
- 引用其他檔案時寫真實路徑，改名/搬家後要全域搜尋修引用（`Grep` 舊路徑）。

## 本專案實例
maintenance.md 教訓紀錄（2026-07-04）：diagnosis.md 與 letter.md 曾對同一件事（fable 模型警示、API key 提醒）各寫一份，且其中一條已解決卻沒清掉——單檔各自看都「沒問題」，跨檔一比就是重複＋過期。所以精簡 audit 必須**跨檔**核對「同一件事是否寫在兩個家」與「警示是否已過期」，不能一次只看一個檔案。本技能庫（skills/）大量引用 rules/ 而不複述，就是為了不重蹈這個坑。
