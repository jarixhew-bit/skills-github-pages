# 技能：數字類產出的核對（記帳、金額、日期、合計）

## 目的
使用者用本 repo 記帳（`expense-tracker.html`）和管交易（`trading/`）。文字錯一句他看得出來；數字錯一個他會直接信，然後基於錯的數做決定。數字是本專案唯一「錯一個就毀掉信任」的產出類型。

## 何時觸發
產出或修改任何包含金額、日期、匯率、合計、筆數的內容時：報表、匯出功能、OCR 預填、記帳彙總、交易訊號。

## 機械步驟
1. **抽驗**：隨機抽 3 筆，逐筆回到原始資料（使用者上傳的單據、原始網頁、交易紀錄）核對金額與日期。「隨機」= 不許只抽開頭 3 筆。
2. **重算**：所有合計、小計、分類彙總，用獨立方法重算一次（原本是程式算的就手工加總抽查；原本手工的就寫一行腳本算）。兩個結果對不上 → 停，找出差異來源才能交付。
3. **貨幣單位**：出現兩種以上貨幣 → 停下來問匯率基準（這是 judgment.md 第 3 節的「必問」情形，因為換算方式改變整份報告）。
4. **自動識別的數字**（OCR、爬蟲抓價）：一律視為「候選值」，不是事實。呈現時要讓使用者能看到原始憑證對照；程式行為上只在欄位為空時預填，不覆蓋人工輸入。
5. **日期格式**：跨頁面統一用 ISO（YYYY-MM-DD）存儲；顯示格式可在地化，但存儲格式不許混。

## 硬規則
- 合計對不上時，禁止「差不多、四捨五入的關係吧」——找到原因或標註「合計未通過核對」再交付。
- 查不到的數字標「未查證」，禁止編一個合理值。沒有例外。

## 本專案實例
`expense-tracker.html` 的 OCR 預填功能（commit `3333968`，#91）：Tesseract.js 本地識別單據後預填金額/日期。設計時遵守的規則就是上面第 4 步——**只在欄位為空時預填、不阻塞保存、不覆蓋人工輸入**。判斷依據：OCR 對中文單據的金額識別會錯，錯的預填如果覆蓋了使用者手填的正確值，等於系統主動製造錯帳。自動化在數字領域的定位永遠是「省打字」，不是「替人決定」。
